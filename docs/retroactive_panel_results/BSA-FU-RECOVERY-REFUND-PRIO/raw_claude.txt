## Findings

### HIGH: apps/platform-domain/bay-session/cmd/committed-bet-recovery/worker_test.go:1163
Description: `TestWorker_VoidNeverDebited_LostStateRace` does not exercise the code path it claims to cover. The test seeds bet 199 as `committed`, then flips `BetState = "recovery_failed"` BEFORE `drainOnce` runs — not "between the claim and the per-action CAS" as the docstring asserts. Because the claim filter selects only `committed` bets, the bet is never claimed and `voidNeverDebited` is never called. The test's own trailing comments admit this ("the fake doesn't expose one, so the simpler assertion is..."). All four assertions (`NoError`, state == "recovery_failed", zero wallet calls) are vacuously true regardless of `voidNeverDebited`'s correctness. The real CAS guard (`affected == 0` branch in `worker.go:431-434`) is not actually tested. A regression that removed the `affected == 0` check or that re-introduced a wallet call on the lost-race path would still pass this test. In a Critical-risk financial flow this is a false-confidence test, which is the exact failure mode the rubric calls out as worse than no test.
Fix: Either (a) add a `TransitionCommittedToRefunded` hook to the fake store (as the deferred-findings section acknowledges is needed) and seed it to return `affected=0` on the first call so the real race path runs, or (b) delete this test outright and rename the deferred-findings note to admit the gap. Do not leave a test whose name promises race coverage that the assertions can't deliver.

### MEDIUM: apps/platform-domain/bay-session/cmd/committed-bet-recovery/worker.go:430
Description: `voidNeverDebited`'s SQL-error branch (Tasker reports 76.9% coverage) is not exercised by any test — same gap as the `LostStateRace` test above. There is no test that drives `TransitionCommittedToRefunded` to return a non-nil error, so the error-log + `actionsTotal{outcome="error"}` increment path is dead from a test perspective. This is acknowledged in deferred findings but it stacks with the HIGH above: the void-never-debited action has zero exercised failure paths.
Fix: Add a fake-store hook (`hookOnTransitionCommittedToRefunded` or equivalent) and one test that drives it to return an error. The same hook unblocks the HIGH-finding fix.

### LOW: apps/platform-domain/bay-session/cmd/committed-bet-recovery/worker.go:323
Description: `RefundReasonNeverDebited = "never_debited"` is a contract that downstream reconciliation tooling must understand. The summary documents that reconciliation must treat this reason as "no wallet row expected", but nothing in this diff verifies the reconciliation side handles the new value. If existing audit queries assume every `refund_reason` has a corresponding wallet `BET_REFUND` row, this PR silently introduces orphaned voids in the audit view.
Fix: Add a tracking ticket (out of scope for this PR) for the reconciliation-side update, and grep the repo for consumers of `smg_bet.refund_reason` before merging to confirm none of them break on the new value.
