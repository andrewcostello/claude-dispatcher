### Blocking findings

- **HIGH** at `apps/platform-domain/bay-session/cmd/committed-bet-recovery/worker_test.go:1163` — `TestWorker_VoidNeverDebited_LostStateRace` does not exercise the code path it claims to cover. The test seeds bet 199 as `committed`, then flips `BetState = "recovery_failed"` BEFORE `drainOnce` runs — not "between the claim and the per-action CAS" as the docstring asserts. Because the claim filter selects only `committed` bets, the bet is never claimed and `voidNeverDebited` is never called. The test's own trailing comments admit this ("the fake doesn't expose one, so the simpler assertion is..."). All four assertions (`NoError`, state == "recovery_failed", zero wallet calls) are vacuously true regardless of `voidNeverDebited`'s correctness. The real CAS guard (`affected == 0` branch in `worker.go:431-434`) is not actually tested. A regression that removed the `affected == 0` check or that re-introduced a wallet call on the lost-race path would still pass this test. In a Critical-risk financial flow this is a false-confidence test, which is the exact failure mode the rubric calls out as worse than no test.
  - *Fix:* Either (a) add a `TransitionCommittedToRefunded` hook to the fake store (as the deferred-findings section acknowledges is needed) and seed it to return `affected=0` on the first call so the real race path runs, or (b) delete this test outright and rename the deferred-findings note to admit the gap. Do not leave a test whose name promises race coverage that the assertions can't deliver.
- **CRITICAL** at `apps/platform-domain/bay-session/cmd/committed-bet-recovery/decide.go:208` — The `ActionVoidNeverDebited` branch assumes that if `debit_confirmed_at IS NULL`, the wallet has no `BET` transaction. In a distributed system, a network timeout or pod crash immediately following a successful `wallet.Bet` RPC (but before the local `debit_confirmed_at` update) leaves the local DB with a NULL confirmed timestamp even though the user's funds were successfully debited. By voiding the bet locally and explicitly skipping the wallet call, this money is permanently trapped in the wallet without a refund, resulting in financial loss for the user.
  - *Fix:* Remove `ActionVoidNeverDebited` and the local void path entirely. To prevent calling `RefundBet` on an unconfirmed bet, simply add `&& !bet.DebitConfirmedAt.IsZero()` to the existing `ActionConfirmRefundResultTimeout` condition. This will cause unconfirmed expired bets to fall through to `ActionConfirmDebitStayCommitted` (which safely and idempotently syncs the debit state with the wallet). On the subsequent worker loop, the bet will have `debit_confirmed_at` set and will correctly and safely process through the timeout-refund branch.
- **CRITICAL** at `apps/platform-domain/bay-session/cmd/committed-bet-recovery/worker.go:430` — `voidNeverDebited` decides from a previously claimed snapshot and then updates the row with a CAS that only checks `bet_state='committed'`. It does not re-check that `debit_confirmed_at` is still `NULL`. If the wallet debit succeeds after the claim but before this update, the row can be transitioned to `REFUNDED` with `refund_reason='never_debited'` even though a real `BET` ledger row now exists and the customer has actually been debited. Because this path intentionally skips `wallet.RefundBet`, that race leaves the wallet debited with no compensating refund while the bet record says refunded.
  - *Fix:* Make the void transition conditional on the debit still being unconfirmed at write time, not just on `bet_state`. Concretely, the `UPDATE`/transition used by `ActionVoidNeverDebited` should include `debit_confirmed_at IS NULL` (or another equivalent expected-version predicate), and a 0-row result should trigger a re-evaluation path rather than being treated as a harmless lost-state race. If the debit has landed, the worker must switch to the normal wallet-backed refund/settlement flow.
