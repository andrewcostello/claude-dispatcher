## Findings

### HIGH: apps/platform-domain/bay-session/cmd/bay-session/stream_target.go:660
Description: Stale-frame drops do not advance `lastSeq`, which produces phantom sequence-gap metric increments under exactly the cross-pod inversion this change is designed to handle. The drain loop now drops a frame via `continue` after the `msgDBVersion <= curDBVersion` test but before `lastSeq.Store(msgSeq)` at line ~691. The next delivered frame's broker_seq is then compared against the OLD `lastSeq`, and `gap := msgSeq - curSeq - 1` registers a non-zero gap even though the broker delivered every frame contiguously. The Tasker explicitly tells SRE in `deploy/bay-session.yaml` (HPA block) that `bay_session_subscriber_sequence_gap_total` is now an "orthogonal" broker-delivery-loss signal — but this code path makes it co-vary 1:1 with `droppedStaleFrames`. After HPA enable, every cross-pod correction will look like a broker-loss event.
Fix: Update `lastSeq.Store(msgSeq)` before the `continue` on the stale-drop path (without running the gap math). The broker DID deliver this frame; the gap accounting must reflect broker truth, not post-dedup truth. Add a regression test that drives a stale drop and asserts `sequenceGap` does NOT increment.

### HIGH: apps/platform-domain/bay-session/cmd/bay-session/stream_target.go:662
Description: The back-compat fallthrough `msgDBVersion > 0 && msgDBVersion <= curDBVersion` silently disables causal dedupe for any frame whose db_version is zero. There is no warning log, no metric, no guard. In a financial system, a future producer path that constructs a `SessionStateUpdate` without going through `publishStationSnapshot` (e.g., a replay tool, a migration job, a copy-paste in a new handler) will bypass the entire correctness contract this ticket is supposed to deliver — and there will be no observable signal that it is happening. The Tasker's summary acknowledges this as deferred-and-untracked: "once all producers are confirmed BSA-FU-XPOD-ORDERING-aware the guard can be tightened in a follow-up." No follow-up ticket is referenced.
Fix: Either (a) drop the `> 0` guard now and fix the test fakes that omit the field — the test churn is small and the production safety gain is large; or (b) keep the guard but add a `bay_session_subscriber_unstamped_frames_total{target_kind}` counter incremented every time the guard rescues a zero-version frame, and file the follow-up ticket explicitly in this PR's deferred-findings block. Without one of these, the contract has a silent escape hatch.

### HIGH: libs/protos/pkg/bay_session/v1/bay_session.proto:57
Description: The monotonicity contract for `db_version` rests entirely on the statement (Tasker summary): "the source of truth the FOR UPDATE row lock serialises against [is `bay_session_state.updated_at`]." That is only true if `updated_at` is assigned inside the lock window using a clock source that is monotonic across pods. The diff does not show the UPDATE statement that assigns `updated_at`. If it uses Postgres `now()` / `current_timestamp` (transaction-start time), two pods serializing on the FOR UPDATE lock can produce non-monotonic `updated_at` values — Pod B that started its transaction earlier but committed later under contention will write an EARLIER `updated_at`, and the consumer will then incorrectly drop B's (newer-by-commit-order) frame as "stale." If `updated_at` is assigned via Go `time.Now()` in app code, inter-pod clock skew (NTP-normal drift is tens of ms) can flip the ordering in the same way. Only `clock_timestamp()` or a sequence-backed monotonic source is safe here. For a financial system, this assumption needs to be proven, not asserted.
Fix: In this PR, either link to or include the SQL that assigns `bay_session_state.updated_at` and demonstrate it uses `clock_timestamp()` (or equivalent). If it uses `now()` / app `time.Now()`, switch to `clock_timestamp()` AND add an integration test that holds the FOR UPDATE lock on Pod A while Pod B's transaction queues, then asserts Pod B's `updated_at` > Pod A's. The current property tests only exercise the consumer-side reorder rule; they assume the producer-side invariant holds.

### MEDIUM: apps/platform-domain/bay-session/cmd/bay-session/stream_target.go:662
Description: The inclusive `<=` dedupe rule drops the LATER frame on a microsecond collision. The Tasker's summary acknowledges this and claims "full-snapshot supersedure means no data loss" — but that property only holds when a subsequent mutation arrives. Under an idle period after a collision (two commits resolving inside the same microsecond, then traffic quiesces), the consumer is stuck with the OLDER causal state until the next mutation. For a UI consuming this stream — particularly anything showing bet state — that's an observable stale-state window. No metric distinguishes "stale drop because broker reordered" (the intended path) from "stale drop because aliasing dropped the newer frame" (the data-loss path).
Fix: Either tighten the watermark to a (db_version, broker_seq) tuple so collisions break by broker_seq, or document that the supersedure assumption requires bounded idle time and add a metric / log line on equal-version drops so the collision rate is observable.

### MEDIUM: apps/platform-domain/bay-session/cmd/bay-session/stream_session_property_test.go:104
Description: The property test deliberately excludes db_version=0 via `rapid.Int64Range(1, 50)`, with the comment that "exercising that path here would conflate the dedupe contract with the bypass path." But the bypass path IS in production code and IS reachable from a misbehaving producer — there is no test anywhere that pins what happens when a mix of zero-version and positive-version frames interleave with the snapshot baseline. Given the back-compat fallthrough exists, its behavior is part of the contract and needs property coverage.
Fix: Add a second property test (or extend this one) that draws db_versions from `Int64Range(0, 50)` and asserts the documented behavior: zero-version frames always deliver, positive-version frames dedupe normally, and zero-version frames never advance the watermark.

### MEDIUM: apps/platform-domain/bay-session/cmd/bay-session/stream_target.go:691
Description: After a stale drop, the heartbeat path will continue to echo the prior `lastSeq` (the broker_seq before the dropped frame), even though the broker has moved on. Operator running `bay_session_subscriber_heartbeat_last_seq` will see a "stuck" heartbeat that doesn't match the current broker tip, then a jump when the next non-stale frame arrives. Not a correctness bug, but it pairs poorly with HIGH #1: under HPA scale-out, operators looking at heartbeats AND sequence_gap will both see misleading signals.
Fix: Advancing `lastSeq` on stale drops (HIGH #1 fix) resolves this too.

### LOW: apps/platform-domain/bay-session/cmd/bay-session/stream_target.go:706
Description: `messageDBVersion` does a nil-check on the adapter hook; `MessageSequence` does not. Inconsistent defensiveness within the same struct. Either both should be nil-safe or neither.
Fix: Drop the nil check from `messageDBVersion` and require all adapters to populate `MessageDBVersion` (the two production adapters already do; only stale test fakes would break, and they should be updated).

## Notes
The shape of this change is right — broker sequence cannot be the causal authority across pods, and stamping a producer-side watermark is the correct fix. My objection is to the execution: the gap-metric regression (HIGH #1) is visible directly in the diff and will materially mislead SRE on the very rollout this ticket enables; the back-compat fallthrough (HIGH #2) is an unguarded escape hatch in a financial system; and the producer-side monotonicity (HIGH #3) is asserted, not demonstrated. The Tasker self-approved under the BSA direct-branch workflow without external review — exactly the case where cross-family review needs to bite. I would re-review on a respin that fixes #1, picks one of the two options for #2, and includes evidence (SQL or integration test) for #3.
