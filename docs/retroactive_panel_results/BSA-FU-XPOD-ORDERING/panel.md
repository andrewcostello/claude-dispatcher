### Blocking findings

- **HIGH** at `apps/platform-domain/bay-session/cmd/bay-session/stream_target.go:660` — Stale-frame drops do not advance `lastSeq`, which produces phantom sequence-gap metric increments under exactly the cross-pod inversion this change is designed to handle. The drain loop now drops a frame via `continue` after the `msgDBVersion <= curDBVersion` test but before `lastSeq.Store(msgSeq)` at line ~691. The next delivered frame's broker_seq is then compared against the OLD `lastSeq`, and `gap := msgSeq - curSeq - 1` registers a non-zero gap even though the broker delivered every frame contiguously. The Tasker explicitly tells SRE in `deploy/bay-session.yaml` (HPA block) that `bay_session_subscriber_sequence_gap_total` is now an "orthogonal" broker-delivery-loss signal — but this code path makes it co-vary 1:1 with `droppedStaleFrames`. After HPA enable, every cross-pod correction will look like a broker-loss event.
  - *Fix:* Update `lastSeq.Store(msgSeq)` before the `continue` on the stale-drop path (without running the gap math). The broker DID deliver this frame; the gap accounting must reflect broker truth, not post-dedup truth. Add a regression test that drives a stale drop and asserts `sequenceGap` does NOT increment.
- **HIGH** at `apps/platform-domain/bay-session/cmd/bay-session/stream_target.go:662` — The back-compat fallthrough `msgDBVersion > 0 && msgDBVersion <= curDBVersion` silently disables causal dedupe for any frame whose db_version is zero. There is no warning log, no metric, no guard. In a financial system, a future producer path that constructs a `SessionStateUpdate` without going through `publishStationSnapshot` (e.g., a replay tool, a migration job, a copy-paste in a new handler) will bypass the entire correctness contract this ticket is supposed to deliver — and there will be no observable signal that it is happening. The Tasker's summary acknowledges this as deferred-and-untracked: "once all producers are confirmed BSA-FU-XPOD-ORDERING-aware the guard can be tightened in a follow-up." No follow-up ticket is referenced.
  - *Fix:* Either (a) drop the `> 0` guard now and fix the test fakes that omit the field — the test churn is small and the production safety gain is large; or (b) keep the guard but add a `bay_session_subscriber_unstamped_frames_total{target_kind}` counter incremented every time the guard rescues a zero-version frame, and file the follow-up ticket explicitly in this PR's deferred-findings block. Without one of these, the contract has a silent escape hatch.
- **HIGH** at `libs/protos/pkg/bay_session/v1/bay_session.proto:57` — The monotonicity contract for `db_version` rests entirely on the statement (Tasker summary): "the source of truth the FOR UPDATE row lock serialises against [is `bay_session_state.updated_at`]." That is only true if `updated_at` is assigned inside the lock window using a clock source that is monotonic across pods. The diff does not show the UPDATE statement that assigns `updated_at`. If it uses Postgres `now()` / `current_timestamp` (transaction-start time), two pods serializing on the FOR UPDATE lock can produce non-monotonic `updated_at` values — Pod B that started its transaction earlier but committed later under contention will write an EARLIER `updated_at`, and the consumer will then incorrectly drop B's (newer-by-commit-order) frame as "stale." If `updated_at` is assigned via Go `time.Now()` in app code, inter-pod clock skew (NTP-normal drift is tens of ms) can flip the ordering in the same way. Only `clock_timestamp()` or a sequence-backed monotonic source is safe here. For a financial system, this assumption needs to be proven, not asserted.
  - *Fix:* In this PR, either link to or include the SQL that assigns `bay_session_state.updated_at` and demonstrate it uses `clock_timestamp()` (or equivalent). If it uses `now()` / app `time.Now()`, switch to `clock_timestamp()` AND add an integration test that holds the FOR UPDATE lock on Pod A while Pod B's transaction queues, then asserts Pod B's `updated_at` > Pod A's. The current property tests only exercise the consumer-side reorder rule; they assume the producer-side invariant holds.
- **HIGH** at `apps/platform-domain/bay-session/cmd/bay-session/stream_target.go:653` — Regression in deduplication logic for legacy producers. By removing the `msgSeq <= cur` check and replacing it with only the `msgDBVersion` check, the consumer no longer dedupes frames from legacy producers (or test fakes) that omit the `db_version` field. JetStream redeliveries or overlapping replays of legacy frames will pass the `msgDBVersion > 0` guard and be processed multiple times, violating the full-snapshot supersedure contract and potentially causing state flickering in the subscriber.
  - *Fix:* Re-introduce the broker-sequence deduplication check (`msgSeq <= curSeq`) as the primary filter for all incoming messages. The causal `db_version` check should be an additional filter that specifically addresses cross-pod reordering for new producers, but basic broker-level sequence monotonicity must still be enforced for all traffic.
- **HIGH** at `apps/platform-domain/bay-session/cmd/bay-session/stream_target.go:402` — The replay/resume path discards the causal baseline by returning `snapshotDBVersion=0` whenever `sinceSeq` is used. After this change, stale-frame suppression depends on `db_version`, not broker sequence. That means a client that reconnects after seeing a newer causal frame can still be replayed an older causal frame with a higher broker sequence and the server will forward it, because `lastDBVersion` starts at zero. This reintroduces the exact cross-pod ordering bug on reconnects, which is a normal production path, so the fix is incomplete.
  - *Fix:* Resume state must carry the last delivered causal watermark, not only the last broker sequence. Add `db_version` to the client resume contract/heartbeat surface and seed `lastDBVersion` from it on replay, or force the reconnect path back through a fresh snapshot so the server can reconstruct a correct causal baseline before forwarding live/replayed messages.
- **HIGH** at `apps/platform-domain/bay-session/cmd/bay-session/session_logic.go:742` — `db_version` is derived from `updated_at.UnixMicro()` and `drainTargetBuffer` drops any frame with `db_version <= lastDBVersion`. Two distinct commits that land in the same microsecond therefore alias to the same watermark, causing the later state to be dropped permanently as a duplicate even though it is newer. In a financial system this is not a harmless optimization: a subscriber can remain on stale state until some later mutation happens, and if no later mutation occurs the stream is wrong indefinitely.
  - *Fix:* Use a strictly monotonic version source instead of wall-clock microseconds, such as a DB sequence/version column that advances on every committed mutation. If the schema cannot change immediately, the dedupe rule cannot safely treat equal timestamps as duplicates; you need an additional tie-breaker that is monotonic per station and preserved across publishers.
