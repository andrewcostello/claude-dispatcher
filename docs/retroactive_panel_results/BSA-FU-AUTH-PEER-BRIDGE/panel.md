### Blocking findings

- **HIGH** at `apps/platform-domain/bay-session/cmd/bay-session/peer_allowlist_test.go:96` — `TestPeerAllowlist_SimulatorHandlerProcedures` is a tautology that gives false confidence. The test iterates a list of RPC names (`RemovePlayer`, `ConfigureSession`, `SetRotationOrder`, `StartHit`, `ProcessShot`, `LeaveSession`, `SetPlayerConfirmationSettings`, ...) and for each name asserts that `requireSimulator(peerCtx, realStation)` rejects a peer principal — but it never actually invokes the handler named in the subtest. Every subtest runs the same single-line assertion against the same helper. Identical structure on the `allowed` side: it asserts `requireSimulatorOrPeer` accepts a peer, regardless of which RPC name is in the subtest. The test would still pass if a future developer "fixed" a permission-denied error on `RemovePlayer` by switching it to `requireSimulatorOrPeer`, or if `LeaveSession` were quietly routed to the peer-allowing helper. On an auth boundary in a financial system, a test that looks like per-procedure coverage but isn't is worse than no test — it lulls reviewers and CI into assuming the call sites are pinned. The Tasker self-review even calls out the "§1 + §2 review finding" this test is supposed to safety-net; it doesn't.
  - *Fix:* Either (a) replace the table-driven test with a parser-based call-site check (e.g. `go/ast` walk over `simulator_handler.go` and `dual_handler.go` asserting that each non-allowlisted method body calls `requireSimulator` and only the allowlisted methods call `requireSimulatorOrPeer`), or (b) drive the actual handler entry points end-to-end with a peer-shaped claims context and assert the response code (`CodePermissionDenied` for denied procedures, success/business-error for allowed). Either form will fail if the call-site wiring drifts. The current shape locks in nothing about the per-RPC surface and should not exist.
- **HIGH** at `apps/platform-domain/bay-session/cmd/bay-session/peer_allowlist_test.go:106` — The `TestPeerAllowlist_SimulatorHandlerProcedures` test provides false confidence. It aims to verify that specific handlers enforce or bypass the peer check, but it merely iterates over an array of hardcoded procedure names (strings) and directly invokes the helper functions (`requireSimulator` and `requireSimulatorOrPeer`). It never actually invokes the RPC handlers it claims to be testing (e.g., `PushFrsUpdate` or `SetActivePlayer`). Consequently, if a developer mistakenly applies the wrong helper in a handler, this test will still pass because it only evaluates the helpers against string literals.
  - *Fix:* Refactor the test to instantiate a `SimulatorHandler` (or the respective dual handler) and explicitly invoke each actual RPC method with the synthesized peer context. Assert that the appropriate gRPC error code (e.g., `connect.CodePermissionDenied` / `authz.ErrWrongStation`) is returned for rejected procedures, and that allowlisted procedures proceed without authz rejection.
- **HIGH** at `apps/platform-domain/bay-session/cmd/bay-session/peer_allowlist_test.go:93` — The new allowlist test does not actually verify the per-RPC authorization surface it claims to protect. Every subtest just calls `requireSimulator` or `requireSimulatorOrPeer` directly, with procedure names used only as labels. That means regressions at the real call sites would still pass: for example, if `SetActivePlayer` were changed back to `requireSimulator`, or if the inline `isSmgCorePeer` guard were removed from `GetSessionState` / `StreamSessionState`, this test suite would remain green. In a high-risk auth change, a test that gives false confidence around the exact RPC allowlist is blocking.
  - *Fix:* Replace this with behavioral tests against the actual handler methods or another mechanism that inspects the real call sites. At minimum, each allowlisted RPC should be exercised through its handler path with peer claims and shown to succeed, and each non-allowlisted RPC should be shown to reject the same peer claims. For the dual-handler inline guards, the test must hit `GetSessionState` and `StreamSessionState` specifically rather than assuming `requireSimulatorOrPeer` covers them.
