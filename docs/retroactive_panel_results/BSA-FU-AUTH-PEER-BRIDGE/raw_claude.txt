## Findings

### HIGH: apps/platform-domain/bay-session/cmd/bay-session/peer_allowlist_test.go:96
Description: `TestPeerAllowlist_SimulatorHandlerProcedures` is a tautology that gives false confidence. The test iterates a list of RPC names (`RemovePlayer`, `ConfigureSession`, `SetRotationOrder`, `StartHit`, `ProcessShot`, `LeaveSession`, `SetPlayerConfirmationSettings`, ...) and for each name asserts that `requireSimulator(peerCtx, realStation)` rejects a peer principal — but it never actually invokes the handler named in the subtest. Every subtest runs the same single-line assertion against the same helper. Identical structure on the `allowed` side: it asserts `requireSimulatorOrPeer` accepts a peer, regardless of which RPC name is in the subtest. The test would still pass if a future developer "fixed" a permission-denied error on `RemovePlayer` by switching it to `requireSimulatorOrPeer`, or if `LeaveSession` were quietly routed to the peer-allowing helper. On an auth boundary in a financial system, a test that looks like per-procedure coverage but isn't is worse than no test — it lulls reviewers and CI into assuming the call sites are pinned. The Tasker self-review even calls out the "§1 + §2 review finding" this test is supposed to safety-net; it doesn't.
Fix: Either (a) replace the table-driven test with a parser-based call-site check (e.g. `go/ast` walk over `simulator_handler.go` and `dual_handler.go` asserting that each non-allowlisted method body calls `requireSimulator` and only the allowlisted methods call `requireSimulatorOrPeer`), or (b) drive the actual handler entry points end-to-end with a peer-shaped claims context and assert the response code (`CodePermissionDenied` for denied procedures, success/business-error for allowed). Either form will fail if the call-site wiring drifts. The current shape locks in nothing about the per-RPC surface and should not exist.

### MEDIUM: apps/platform-domain/bay-session/cmd/bay-session/smg_core_peer_verifier.go:206
Description: Successful peer-token acceptance is logged at `Debug` only, carrying just `subject` and `jti`. Peer bypass-allowlisted RPCs (`SetActivePlayer`, `SetFrsMode`, `PushFrsUpdate`) mutate session state, and the inner-error-passthrough design (anti-probing — every rejection surfaces as `ErrTokenSignature`) means a defender investigating a suspicious peer-bridge acceptance at 3am will have no audit breadcrumb above Debug. The deny-list failure path *does* log at Error, which is correct; only the success path is under-observable.
Fix: Promote the acceptance log to `Info` and include the verified `iss` and `aud` (not just `sub` / `jti`). This is a low-volume code path (handful of RPCs/sec at most) so log volume isn't a concern, and it gives forensics a positive record of "we accepted a peer token from issuer X for subject Y at time T".

### LOW: apps/platform-domain/bay-session/cmd/bay-session/smg_core_peer_verifier.go:230
Description: The synthesized `*authz.Claims` stamps `Subject`, `Issuer`, and `ID` (jti) but not `Audience`, even though the bridge has just verified the audience matches `v.expectedAudience`. Tasker flagged this as a deferred finding. Not a correctness issue today, but if any downstream audit-log enricher ever serialises the verified audience, it will be missing for peer principals.
Fix: After audience check passes, stamp `claims.Audience = jwt.ClaimStrings{v.expectedAudience}` on the synthesized RegisteredClaims. One line, no behavioural change.

### LOW: apps/platform-domain/bay-session/cmd/bay-session/simulator_handler.go:68
Description: `requireSimulatorOrPeer(ctx, stationID)` accepts `stationID` but for peer principals never consults it (the body — not shown in this diff hunk but inferable from the tests — short-circuits on `isSmgCorePeer(claims)`). The requested station for a peer call is then never surfaced anywhere — not in logs, not in metrics. A peer principal that asks for `station_id=999999` is indistinguishable from one asking for a real station in this codepath.
Fix: Either log the requested station alongside the peer-acceptance log line (see MEDIUM finding above), or have `requireSimulatorOrPeer` emit a metric tag like `peer_bridge_requests{station_id=...}`. Optional but cheap; defensible to defer.

## Notes
The cryptographic posture is genuinely strong: alg-locked RS256, iss + aud + jti + deny-list all in the same gate, fail-closed on Redis error, anti-probing via inner-error passthrough, constructor-level invariants for iss/denyList. The narrowed five-RPC allowlist also closes a real privilege overreach. My block is the test in `peer_allowlist_test.go` — on this part of the codebase a test that *looks* like it pins the allowlist but doesn't is exactly the kind of artifact that lets the next regression slip through code review. Fix that and stamp the verified audience and this is a clean approve. Per-dimension Security/Resilience are reference-quality; Maintainability is held down purely by the test artifact.
