## Findings

### MEDIUM: apps/platform-domain/bay-session/cmd/committed-bet-recovery/worker.go:451
Description: The skip-wallet branch (when `bet.DebitConfirmedAt.IsZero()` is false) silently bypasses the wallet RPC with no log emission. An operator triaging "why did this bet settle without a wallet.Bet line in the audit log?" has to infer it from the absence of the wallet call rather than seeing a positive signal. Given this codepath is post-incident-recovery and the audit trail is the primary forensic surface, a one-line debug/info log ("skipping wallet.Bet — debit already confirmed at <timestamp>") would close the loop. Not blocking because the structured "recovery: confirmed debit + advanced to settling" log at the end captures the final state, but operators reading from the wallet side will see a gap.
Fix: Add a single `w.logger.WithContext(ctx).Debug("recovery: skipping wallet.Bet — debit already confirmed", slog.Int64("bet_id", bet.BetID), slog.Time("debit_confirmed_at", bet.DebitConfirmedAt))` immediately before the implicit "skip" branch (i.e., as the else of the `IsZero()` check, or as a guard log just before line 451).

### MEDIUM: apps/platform-domain/bay-session/cmd/committed-bet-recovery/worker.go:619
Description: `payout_amount_projected` is computed and logged here but explicitly NOT persisted to `smg_bet` (per summary, BSA-7-2 recomputes downstream). If BSA-7-2 recomputes via a path that doesn't invoke `outcome.Compute` — e.g., reads stored `multiplier` and does its own arithmetic — there's no test or invariant pinning that the two paths produce equal results. The current commit doesn't introduce drift (BSA-7-2 only sees the now-correct multiplier on disk), but the log's "projected" value will be unverifiable against the ultimately-persisted value without a cross-check.
Fix: Either persist the projected payout as a column on `smg_bet` here (so downstream can CHECK it) OR add a comment-pointer in BSA-7-2's payout-compute path that it MUST call `outcome.Compute` to stay bit-identical. The latter is cheaper and matches the leaf-package architecture established by this change.

### LOW: apps/platform-domain/bay-session/cmd/committed-bet-recovery/worker.go:454
Description: `defer cancel()` is declared inside the `if bet.DebitConfirmedAt.IsZero()` block. In Go, deferred calls execute at function return, not block exit — so the wallet-call's timeout context lingers across the entire settlement-write block (hit-read, compute, transition). The context is unused past the wallet call, so this is purely a resource-cleanup-timing nit, not a correctness issue.
Fix: Wrap the wallet RPC + mark-debit in an inline anonymous function or extract to a method so the `defer cancel()` fires at the wallet-call boundary. Optional polish.

### LOW: apps/platform-domain/bay-session/internal/outcome/outcome.go:106
Description: The property test `TestCompute_PayoutIsAmountTimesMultiplierDiv100` caps `amount` at 1e9, which never exercises the int64 multiplication boundary close to `BetAmountOverflowGuard × top_multiplier`. The single explicit `amount over guard` test pins rejection of the +1 case but doesn't pin the *just-below-overflow* arithmetic (e.g., `amount = BetAmountOverflowGuard, multiplier = 300` → `payout = 3e16`).
Fix: Add a single-case test asserting `Compute(0, BetAmountOverflowGuard)` succeeds (lose path, no overflow) and `Compute(48, BetAmountOverflowGuard)` returns the expected win-path payout without overflow. Pins the guard semantics at the boundary the `init()` invariant defends.
